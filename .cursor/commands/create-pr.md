# PR を作成

## 概要
適切な説明・ラベル・レビュアーを含む、よく整理されたプルリクエストを作成する。

## 手順

### 1. ブランチを準備
- すべての変更がコミットされていることを確認
- ブランチをリモートにプッシュ
- ブランチが main と最新同期されていることを確認
- コンフリクトがないことを確認

### 2. PR の説明を書く
- **変更点を明確に要約**: 何を変更したか、なぜ変更したかを簡潔に説明
- **背景や目的を含める**: 問題点や要件を説明
- **破壊的変更があれば列挙**: API変更、データベース変更など
- **UI の変更があればスクリーンショットを追加**: ビフォー・アフターの画像
- **テスト方法を記載**: レビュアーが確認できるように

### 3. PR を設定
- **タイトル**: 内容が伝わる明確なタイトルで PR を作成
  - 例: `feat: ユーザー認証機能を追加`
  - 例: `fix: ログインエラーの修正`
- **適切なラベルを追加**: `feature`, `bugfix`, `documentation` など
- **レビュアーをアサイン**: 関連するチームメンバーを指定
- **関連する issue をリンク**: `Closes #123` など

## PR テンプレート

```markdown
## 変更内容
- [ ] 機能 A を追加
- [ ] バグ修正 B
- [ ] リファクタリング C

## テスト
- [ ] ユニットテスト合格
- [ ] 統合テスト合格
- [ ] 手動テスト完了
- [ ] パフォーマンステスト完了

## チェックリスト
- [ ] コードレビューを受けた
- [ ] ドキュメントを更新した
- [ ] 破壊的変更がない（または明記した）
- [ ] セキュリティチェックを実施した
```

## ベストプラクティス

- **小さなPR**: 1つのPRで1つの機能や修正に集中
- **明確なコミットメッセージ**: 各コミットの意図が明確
- **CI/CDの確認**: すべてのテストとチェックが合格していることを確認
- **コードカバレッジ**: 新しいコードにテストが含まれていることを確認
